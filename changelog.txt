[2026-05-01] WorkoutsScreen — Critical Bug-Squashing Session
  Scope: CreatePlanDialog.kt, RoutineWizardScreen.kt, WorkoutsViewModel.kt
  Author: Antigravity AI Agent (Claude Sonnet)

  Bug 1 — Nested BottomSheet Gesture Collision (Unclickable Tooltips):
    Root cause: InfoTooltip was rendered as a ModalBottomSheet nested inside
    the CreatePlanBottomSheet/RoutineWizardScreen ModalBottomSheet. M3 routes
    all pointer events to the topmost sheet overlay; the parent sheet consumed
    touch events before the child sheet's Info IconButton could receive them.
    Fix: Deleted tooltipSheetState. Replaced the nested ModalBottomSheet with
    an AlertDialog (containerColor = ShadowGrey, RoundedCornerShape(16.dp)).
    AlertDialog uses a different system window layer than BottomSheet, so no
    gesture conflict occurs. Single "Got it" TextButton dismisses it.
    Affects: CreatePlanDialog.kt, RoutineWizardScreen.kt.

  Bug 2 — Highlander Bug Regression (New Plans Default isActive = true):
    Root cause: CreatePlanUseCase has isActive: Boolean = true as its default
    parameter. WorkoutsViewModel.createPlan() never passed isActive explicitly,
    so every new plan was born active — bypassing setPlanAsActive() logic.
    Fix: In WorkoutsViewModel.createPlan(), snapshot the current UiState before
    setting it to Loading. If plans list is non-empty → isActive = false.
    If plans list is empty (first-ever plan) → isActive = true.

  Bug 3 — IME Jitter (BottomSheet keyboard jump on first keystroke):
    Root cause: The content Column inside both ModalBottomSheets had both
    navigationBarsPadding() AND imePadding() applied. M3 ModalBottomSheet
    already manages windowInsets (navigation bars) internally. Double-applying
    insets caused the sheet height to be recalculated twice per frame when the
    keyboard appeared — first from the sheet's own inset adjustment, then from
    the explicit imePadding() — producing a one-frame jitter.
    Fix: Removed navigationBarsPadding() from the content Column.
    Moved imePadding() to be the last modifier before padding(), after
    verticalScroll(), so inset compensation only triggers one layout pass.
    Affects: CreatePlanDialog.kt, RoutineWizardScreen.kt content Columns.

  New String Keys (en/es/ca):
    workouts_tooltip_got_it    ("Got it")

[2026-05-01] Single Active Plan — State Management Bug Fixes
  Scope: WorkoutsViewModel.kt, WorkoutsScreen.kt
  Author: Antigravity AI Agent (Claude Sonnet)

  Bug 1 — Highlander Bug (Multiple Active Plans):
    Root cause: The previous onSetActive lambda called savePlanEdits() with only
    the target plan's isActive flipped. savePlanEdits() maps state.plans, only
    replacing the one plan whose ID matches — leaving all other previously-active
    plans with isActive = true in the local state and on the backend.
    Fix: New dedicated ViewModel function setPlanAsActive(planId: String).
    Logic:
      1. Snapshot currentState before any mutation to capture plansToDeactivate
         (plans where isActive == true AND id != planId).
      2. Optimistic local update: state.plans.map { plan.copy(isActive = plan.id == planId) }
         — a single expression that atomically deactivates ALL plans except the
         target and activates the target.
      3. Backend sync in a coroutine: first deactivate all previously-active plans
         (forEach { updatePlanUseCase(..., isActive = false) }), then activate
         the target. Canonical response from the activate call is merged back.

  Bug 2 — Stale UI (Recomposition Failure):
    Root cause: The old onSetActive = { viewModel.savePlanEdits(selectedPlan.copy(isActive=true)) }
    passed a plan whose isActive was already false in the composable scope. The
    optimistic update in savePlanEdits() only replaced that one element in the
    list. However the lambda also captured selectedPlan by reference at the last
    composition — if the list equals check passed due to object identity being
    reused (Kotlin's inline copy semantics), StateFlow's distinctUntilChanged
    operator silently dropped the emission.
    Fix: Using the new setPlanAsActive(selectedPlan.id), the optimistic update
    maps EVERY element in the list (producing a structurally new list regardless
    of which plans changed), guaranteeing that WorkoutsUiState.Success.equals()
    returns false and StateFlow emits the updated state to all collectors.

  Files changed:
    WorkoutsViewModel.kt  — new setPlanAsActive() function added (lines after savePlanEdits)
    WorkoutsScreen.kt     — onSetActive lambda updated to call setPlanAsActive()

[2026-05-01] WorkoutsScreen UX Overhaul — QA Bug Fix Pass
  Scope: CreatePlanDialog.kt, RoutineWizardScreen.kt, RoutineWizardViewModel.kt,
         WorkoutsScreen.kt, strings.xml (en/es/ca)
  Author: Antigravity AI Agent (Claude Sonnet)

  Bug #1 — Accidental BottomSheet Dismissal (Data Loss Risk):
    Root cause: confirmValueChange not set; onDismissRequest fired immediately
    on outside tap or downward swipe with no gate.
    Fix — CreatePlanBottomSheet: Tracks hasDirtyState (name/description/date
    changed). confirmValueChange = { value -> if block swipe to Hidden when
    dirty }. onDismissRequest shows AlertDialog ("Discard changes?") with
    "Keep Editing" (dismiss dialog) and "Discard" (onDismiss) actions.
    Fix — RoutineWizardScreen ModalBottomSheet in WorkoutsScreen.kt: Same
    pattern using routineWizardViewModel.uiState.hasDirtyState (name not blank
    OR exercises not empty). confirmValueChange + discardDialog shown via
    showRoutineDiscardDialog local state.

  Bug #2 — Crash on Exercise Selection (Step 3):
    Root cause: In the refactored RoutineWizardScreen, the wizard content
    Column was given `.verticalScroll(rememberScrollState())`. WizardStep3Exercises
    contains a LazyColumn with `.fillMaxSize()`. Nesting a LazyColumn inside a
    verticalScroll parent is illegal in Compose — the LazyColumn cannot resolve
    its intrinsic height and throws an IllegalStateException at layout time.
    Fix: Removed `.verticalScroll` from the wizard content Column in
    RoutineWizardScreen. The ModalBottomSheet handles its own scrolling natively.
    WizardStep3Exercises fillMaxSize Columns replaced with fillMaxWidth to be
    safe in both scroll and non-scroll parents.

  Bug #3 — Blank Schedule UI for Cycle/Weekly Plans:
    Root cause: Same as Bug #2. The verticalScroll Column measured all children
    with infinite height constraint. The horizontalScroll Rows inside
    WeeklyDaysSelector/CycleDaysSelector reported 0 measured height under that
    constraint, making them invisible. WeeklyDaysSelector was also receiving
    a zero-height Box container from the AnimatedContent.
    Fix: Removing verticalScroll (Bug #2 fix) resolves this. Added an explicit
    ReadOnlyInheritedScheduleCard fallback in WizardStep2Schedule for any
    unhandled planInterval state to guarantee the schedule section always
    renders something meaningful.

  Bug #4 — "Link to Plan" Defaults to First Plan Instead of Active Plan:
    Root cause: RoutineWizardViewModel.resetWizard() always calls
    cachedPlans.firstOrNull() regardless of which plan the user was viewing.
    Fix — RoutineWizardViewModel: resetWizard(preferredPlanId: String? = null)
    now resolves: preferredPlanId match → active plan fallback → first plan.
    Fix — WorkoutsScreen.kt: LaunchedEffect passes selectedFocusPlanId to
    routineWizardViewModel.resetWizard(preferredPlanId = selectedFocusPlanId).
